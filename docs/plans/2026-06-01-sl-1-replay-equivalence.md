### Task 2: Add the replay CLI command

**Intent:** Expose the existing replay engine as a `replay` subcommand that rebuilds canonical state from an events log into a caller-specified scratch database.
**Acceptance criteria:**
- `fakoli-state replay --from-events <events.jsonl> --into <db>` reconstructs canonical state at the `--into` path from the given events file, reusing the existing `replay_from_empty` engine (no reimplementation of replay logic).
- The command refuses to target the project's live `state.db` and exits non-zero with a clear message, since replay deletes its target.
- A missing or unreadable `--from-events` file produces a clear non-zero exit, not a traceback.
- A CLI test exercises a successful replay into a temp path and the live-DB-guard rejection.
**Scope:** plugins/fakoli-state/bin/src/fakoli_state/cli/replay.py, plugins/fakoli-state/bin/src/fakoli_state/cli/__init__.py, plugins/fakoli-state/tests/test_cli.py

### Task 3: Implement `serialize_state` (canonical-state snapshot)

**Intent:** Provide a single pure function that turns a backend's canonical state into a deterministic, comparable representation — the semantic definition of "the state the system cares about."
**Acceptance criteria:**
- A new `serialize_state(backend)` returns a JSON-serialisable structure covering every canonical collection: project, PRD, features, tasks, all claims, reviews, evidence, and sync mappings.
- Output is fully deterministic: collections are sorted by stable key and object keys are ordered, so repeated calls on the same backend produce byte-identical JSON.
- The function performs reads only — no writes, no clock access, no filesystem access beyond the backend.
- A unit test asserts the function is stable across repeated calls and reflects released/stale claims and reviews (proving it uses the Task 1 methods, not the active-only ones).
**Scope:** plugins/fakoli-state/bin/src/fakoli_state/state/snapshot.py, plugins/fakoli-state/tests/test_snapshot.py
**Agent:** guido
**Verify:** `uv run --project plugins/fakoli-state/bin --with pytest pytest plugins/fakoli-state/tests/test_snapshot.py -q`
**Depends on:** Task 1

### Task 4: Golden fixture + equivalence and idempotence tests

**Intent:** Prove that replaying a non-trivial event log reproduces the canonical state produced by normal operation, and that replay is deterministic.
**Acceptance criteria:**
- A committed fixture `events.jsonl` exercises a realistic project: project init, multiple tasks, a claim with lease and heartbeat, a renew, a review acceptance, a stale-claim reap, and a sync mapping — and includes at least one abort/idempotent-no-op line to prove those are skipped on replay.
- A committed `expected-state.json` golden snapshot is the `serialize_state` output for that fixture, generated by a documented, repeatable helper (regeneration is a deliberate human step, not automatic).
- The equivalence test builds the "original" state by applying the fixture events through the normal path and the "replayed" state via the replay engine, and asserts the two `serialize_state` outputs are equal AND equal to the committed golden snapshot.
- The idempotence test replays the same events into two independent scratch databases and asserts the two snapshots are byte-identical.
- The replayed scratch databases live under a temp directory and are never written into the repo (no tracked `.db` files).
**Scope:** plugins/fakoli-state/tests/fixtures/replay/sample-project/events.jsonl, plugins/fakoli-state/tests/fixtures/replay/sample-project/expected-state.json, plugins/fakoli-state/tests/test_replay_equivalence.py
**Agent:** guido
**Verify:** `uv run --project plugins/fakoli-state/bin --with pytest pytest plugins/fakoli-state/tests/test_replay_equivalence.py -q`
**Depends on:** Task 2, Task 3
