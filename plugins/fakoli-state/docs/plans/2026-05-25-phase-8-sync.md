**Goal**: Ship the sync surface for fakoli-state. Multi-provider abstraction (so GitHub Issues, GitHub Projects, Monday, Linear, Jira can all plug in) with GitHub Issues as the first concrete v0 implementation. Bidirectional, polling-based, conflict-resolved, with a reconciliation engine that cross-checks SQLite state vs filesystem vs git.

The spec at `docs/specs/2026-05-24-fakoli-state-v0.md` § Phase 8 said "GitHub Issues only in v0; multi-backend deferred." ## Architecture

### SyncProvider Protocol

```python
class SyncProvider(Protocol):
    """Abstract bidirectional sync target. Implementations: GitHub Issues
    (v1.8.0), GitHub Projects / Monday / Linear / Jira (future).
    """

    provider_id: str          # e.g. "github-issues", "monday", "linear"
    display_name: str         # e.g. "GitHub Issues"

    def push_task(self, task: Task, mapping: SyncMapping | None) -> ExternalRef: ...
    def fetch_task(self, external_id: str) -> ExternalTask | None: ...
    def list_tasks(self) -> list[ExternalTask]: ...
    def delete_task(self, external_id: str) -> None: ...
    def health_check(self) -> ProviderHealth: ...
```

`ExternalTask` and `ExternalRef` are Pydantic models for what the provider returns (id, title, body, status mapping, labels, assignees, url, last_modified).

### RecordedSyncProvider (test double)

Same pattern as `RecordedLLMProvider` (Phase 7). Keyed dict of canned responses. Used in all integration tests so the suite never touches real GitHub.

### Provider registry

```python
PROVIDER_REGISTRY: dict[str, type[SyncProvider]] = {
    "github-issues": GitHubIssuesProvider,
    # contributor-added: monday, linear, jira, ...
}
```

`fakoli-state sync <provider>` resolves the name → class. External providers register by importing their module and calling `register_sync_provider("foo", FooProvider)`.

### SyncMapping (per task)

Already in the spec at line 174. SQL table: `task_id PRIMARY KEY, external_system, external_id, last_synced_at, sync_state, conflict_resolution_strategy`. Phase 8 adds the DDL + handler + payload model.

### Conflict resolution strategies

- `local_wins` — overwrite remote with local on every push
- `remote_wins` — overwrite local with remote on every fetch
- `prompt` — surface to user via CLI prompt; default behavior in interactive contexts
- `manual_merge` — write both sides to a `.sync-conflict-{task_id}.md` file, refuse to sync until resolved

Configured per-task via `SyncMapping.conflict_resolution_strategy`; default falls back to project-level config in `.fakoli-state/config.yaml`.

## Notes for downstream waves

- **Pattern alignment**: SyncProvider Protocol shape MUST mirror LLMProvider Protocol shape (kw-only args, single exception type to catch, Recorded test double, registry pattern). Consistency is the user-facing contract.
- **State events**: every sync mutation (push, fetch, conflict, mapping change) emits a `sync.*` event. Replay-from-empty must reconstruct all SyncMapping rows.
- **Backward compatibility**: existing CLI surface is unchanged. `fakoli-state sync` is NEW; nothing pre-existing moved or changed semantics.
- **Critic preferences**: based on PR #44/#45/#47/#48 review history, this codebase critic flags: (a) duplicate model_config assignments, (b) lazy import comments that aren't actually lazy, (c) token math no-ops, (d) docstrings that lie about parameters, (e) any place a Protocol-conforming provider's bug could abort a "best-effort" wrapping loop. Get those right the first time.
